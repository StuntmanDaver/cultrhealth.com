API endpoints (Plan 02):
```
POST /api/portal/send-otp
  Body: { phone: string }
  Success: { success: true, phone: string }  // phone in E.164
  Errors: 400 (invalid phone), 429 (rate limited), 500 (Twilio error)

POST /api/portal/verify-otp
  Body: { phone: string, code: string }
  Success (patient found): { success: true, hasPatient: true, knownPhone: true, redirect: '/portal/dashboard' }
  Success (known phone, no patient): { success: true, hasPatient: false, knownPhone: true, redirect: '/intake' }
  Success (never-seen phone): { success: true, hasPatient: false, knownPhone: false }  // NO redirect
  Errors: 400 (invalid input), 401 (wrong/expired code), 429 (rate limited)

POST /api/portal/refresh
  No body (reads refresh cookie)
  Success: { success: true }
  Errors: 401 (no token / expired)

POST /api/portal/logout
  No body
  Success: { success: true }
```

        const digits = value.replace(/\D/g, '').slice(0, 10)
        if (digits.length === 0) return ''
        if (digits.length <= 3) return `(${digits}`
        if (digits.length <= 6) return `(${digits.slice(0, 3)}) ${digits.slice(3)}`
        return `(${digits.slice(0, 3)}) ${digits.slice(3, 6)}-${digits.slice(6)}`

    **Create `app/portal/layout.tsx`:**

    This layout wraps all `/portal/*` routes (except login). It provides:
    1. Auth guard: checks for valid portal session, redirects to /portal/login if unauthenticated
    2. Activity-based session refresh: tracks user activity and refreshes access token before expiry

    Implementation:
    ```
    'use client'
    ```

    The layout checks auth on mount and sets up activity tracking.

    **Auth guard logic:**
    - On mount, call `/api/portal/refresh` to verify session is valid
    - If refresh returns 401, redirect to `/portal/login`
    - If refresh returns 200, session is valid -- render children
    - Show a loading spinner during the auth check

    **Activity-based token refresh (AUTH-04):**
    - Track activity events: mousemove, mousedown, keydown, scroll, touchstart (passive listeners)
    - Use a debounced timer: when activity detected, reset a 12-minute timeout
    - When the 12-minute timeout fires (meaning user has been active but token is nearing 15-min expiry), call POST /api/portal/refresh
    - If refresh fails (401), redirect to /portal/login with query param `?expired=true`
    - Use a refresh lock (useRef<Promise>) to prevent concurrent refresh calls
    - Multi-tab: since cookies are shared, any tab's refresh extends all tabs (no special coordination needed)

    **Layout structure:**
    - Minimal for now (Phase 2 will add sidebar/header for dashboard)
    - `min-h-screen bg-brand-cream` wrapper

    - No site header/footer (LayoutShell handles this via HIDE_CHROME_PREFIXES)
