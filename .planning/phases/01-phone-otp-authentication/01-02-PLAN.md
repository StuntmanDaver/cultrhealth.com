  <files>app/api/portal/send-otp/route.ts, app/api/portal/verify-otp/route.ts, tests/api/portal-send-otp.test.ts, tests/api/portal-verify-otp.test.ts</files>
  <behavior>
    Send OTP endpoint (POST /api/portal/send-otp):
    - Accepts JSON body: { phone: string }
    - Validates phone using isValidPhoneNumber() -- returns 400 if invalid
    - Rate limits by IP (5 requests per hour, prefix 'otp-send')
    - Rate limits by phone (5 requests per hour, prefix 'otp-send-phone')
    - Converts phone to E.164 via formatPhoneNumber()
    - Calls Twilio Verify verifications.create({ channel: 'sms', to: phoneE164 })
    - Returns { success: true, phone: phoneE164 } on success
    - Maps Twilio error code 60203 to 429 "Too many OTP requests. Please wait a few minutes."
    - Maps Twilio error code 20404 to 400 "Please enter a valid US phone number."
    - Returns 500 "Failed to send verification code" for other Twilio errors
    - Does NOT reveal whether the phone is registered (security: prevents enumeration)

    Verify OTP endpoint (POST /api/portal/verify-otp):
    - Accepts JSON body: { phone: string, code: string }
    - Validates phone and code (code must be 6 digits) -- returns 400 if invalid
    - Rate limits by IP (10 requests per hour, prefix 'otp-verify')
    - On staging (NEXT_PUBLIC_SITE_URL contains 'staging'), accepts code '123456' without Twilio check
    - Otherwise calls Twilio Verify verificationChecks.create({ code, to: phoneE164 })
    - If check.status !== 'approved': returns 401 with specific error messages:
      - Twilio error 60200: "That code didn't match. Please check and try again."
      - Twilio error 60202: "Too many attempts. Please request a new code."
      - Default: "Invalid or expired code. Please try again."
    - On successful verification:
      1. Calls getPatientByPhone(phoneE164) to resolve Asher Med patient
      2. Also checks local DB via getPortalSessionByPhone(phoneE164) for cached patient ID
      3. Creates access + refresh tokens via createPortalAccessToken/createPortalRefreshToken
      4. Sets cookies via setPortalCookies()
      5. Calls upsertPortalSession() to cache/update the phone-to-patient mapping
      6. Returns { success: true, hasPatient: boolean, redirect: '/portal/dashboard' | '/intake' }
    - If no Asher Med patient AND no cached patient ID: returns hasPatient=false, redirect='/intake'
    - If patient found (Asher Med or cached): returns hasPatient=true, redirect='/portal/dashboard'
  </behavior>

  <behavior>
    Refresh endpoint (POST /api/portal/refresh):
    - Reads refresh token from 'cultr_portal_refresh' cookie on the request
    - If no refresh cookie: returns 401 { error: 'No refresh token' }
    - Calls verifyPortalRefreshToken(refreshToken) -- if null, returns 401 { error: 'Session expired' }
    - Creates new access token via createPortalAccessToken(session.phone, session.asherPatientId)
    - Sets only the access token cookie (new response cookie with maxAge=900, same attributes)
    - Returns 200 { success: true }

    Logout endpoint (POST /api/portal/logout):
    - Calls clearPortalCookies() to delete both portal cookies
    - Returns 200 { success: true }
    - Works even if no cookies are present (idempotent)
  </behavior>
  <action>
    **Create `app/api/portal/refresh/route.ts`:**

    This route is called by the client-side activity tracker when the access token is nearing expiry (12 minutes). It uses the refresh token (which has a 7-day lifetime) to issue a fresh access token.

    Import verifyPortalRefreshToken, createPortalAccessToken, PORTAL_ACCESS_COOKIE, PORTAL_REFRESH_COOKIE from portal-auth.
    Import NextRequest, NextResponse from next/server.

    POST handler:
    1. Read refresh token from `request.cookies.get(PORTAL_REFRESH_COOKIE)?.value`
    2. If missing, return 401
    3. Verify refresh token -- if null (expired or invalid), return 401 with `{ error: 'Session expired. Please log in again.' }`
    4. Create new access token with same phone + patientId from the refresh token session
    5. Create NextResponse with { success: true }
    6. Set access token cookie on the response: `response.cookies.set(PORTAL_ACCESS_COOKIE, newAccessToken, { httpOnly: true, secure: process.env.NODE_ENV === 'production', sameSite: 'lax', maxAge: 900, path: '/' })`
    7. Return response

    Note: The refresh token itself is NOT rotated (static refresh tokens per user decision / research recommendation). This simplifies implementation and avoids token family tracking complexity.

    Add `export const dynamic = 'force-dynamic'`.

    **Create `app/api/portal/logout/route.ts`:**

    Import clearPortalCookies from portal-auth.
    Import NextResponse from next/server.

    POST handler:
    1. Call clearPortalCookies()
    2. Return { success: true }

    Add `export const dynamic = 'force-dynamic'`.
