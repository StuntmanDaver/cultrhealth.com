3. **Gotchas — read these first** (exact heading: `## Gotchas — read these first`). Numbered list, in this order:

   1. **Auth is `Basic`, NOT `Bearer`. AND a second header is required.** Both:
      ```
      Authorization: Basic <HEALTHIE_API_KEY>
      AuthorizationSource: API
      ```
      Anything else returns 401. The API key value goes directly after `Basic ` — do NOT base64-encode it (it's already in API-key form, not username:password). Source: `lib/healthie/client.ts:67-68`.

   2. **Webhooks use RFC 9421 (HTTP Message Signatures), not raw HMAC-SHA256.** Three headers — `Content-Digest`, `Signature-Input`, `Signature` — must be combined per RFC 9421 to compute the signature base. `lib/healthie/webhooks.ts` already implements this; do not roll a new HMAC routine. Read the raw body via `request.text()` BEFORE any other body access; Next.js consumes the body on first read.

   3. **`timingSafeEqual` requires equal-length buffers.** Always check `providedBuf.length === secretBuf.length` first; mismatched lengths throw `TypeError`. The repo's helper does this guard at `lib/healthie/webhooks.ts:138`.

   4. **GraphQL mutations return nested under the mutation name.** Pattern: `{ data: { createClient: { user: {...} } } }`. The `healthieRequest` wrapper takes a `resultKey` parameter — pass the mutation name (e.g. `'createClient'`), and the wrapper extracts to `result.user`. Wrap your Zod schema accordingly: `z.object({ user: HealthieUserSchema }).passthrough()`. Source: `lib/healthie/mutations.ts:28-43`.

   5. **`isHealthieConfigured()` graceful no-op pattern.** All Healthie callers (`ensureHealthiePatient`, `pushLabResultsToHealthie`, etc.) check `isHealthieConfigured()` first and return `null` (NOT throw) if `HEALTHIE_API_KEY` is unset. This lets v2.0 cutover work even before Healthie's API key is provisioned. Do not throw on missing env in integration callers.

   6. **HIPAA log discipline.** Log only error messages and Zod issue paths — never variable values, response bodies, or patient data. The `client.ts` wrapper enforces this:
      ```typescript
      console.error('Healthie GraphQL errors:', json.errors.map(e => e.message))
      console.error('Healthie response validation failed:', parsed.error.issues.map(i => ({
        path: i.path, code: i.code, message: i.message,
      })))
      ```
      Skill consumers MUST follow this pattern; CULTR is HIPAA-compliant and the dev hooks audit for `console.log` near patient identifiers.

   7. **Always pass `skipped_email: true` when creating clients via API.** CULTR sends its own welcome email; if you don't suppress, members get duplicate "Welcome to Healthie" emails. Source: `lib/healthie/patient-sync.ts:32`.

   8. **Subscriptions use ActionCable, NOT graphql-ws.** If a future feature needs realtime (e.g., conversation messages), Healthie's WebSocket protocol is Rails ActionCable — `graphql-ws` and `subscriptions-transport-ws` clients won't connect. Most server-to-server work uses webhooks instead, which is the recommended path.

   9. **Default `HEALTHIE_API_URL` points to STAGING.** Without an explicit env value, code hits `https://staging-api.gethealthie.com/graphql`. Production deploys MUST set `HEALTHIE_API_URL=https://api.gethealthie.com/graphql` explicitly; do not assume the prod env defaults correctly.

   10. **30-second client timeout.** `healthieRequest` aborts after 30s via `AbortController`. Long-running mutations (e.g., bulk form imports) need either a custom timeout or a job-queue pattern; do not block a webhook handler on a 60s+ Healthie call.

5. **Webhook verification** (heading: `## Webhook verification`).
   - Three required headers from Healthie:
     - `Content-Digest: sha-256=:<base64-hash>:`
     - `Signature-Input: sig1=("@method" "@path" "@query" "content-digest" "content-type" "content-length");created=<ts>;keyid="..."`
     - `Signature: sig1=:<base64-hmac>:`
   - Verification flow (4 numbered steps):
     1. `const rawBody = await readWebhookBody(request)` — must be FIRST body read
     2. `const verified = verifyHealthieWebhook(request, rawBody)` — RFC 9421 verification, see `lib/healthie/webhooks.ts:33-67`
     3. `if (!verified) return new Response('invalid signature', { status: 401 })`
     4. `const event = parseWebhookBody(rawBody)` — typed `HealthieWebhookEvent` shape
   - Local-dev fallback: shared-secret via query param `?secret=...` or header `x-healthie-webhook-secret`. Length-checked `timingSafeEqual`. NEVER ship this fallback enabled in production — gate on `process.env.NODE_ENV !== 'production'`.

6. **When implementing a new Healthie call** (heading: `## When implementing a new Healthie call`). Numbered list:

   1. **Use `healthieRequest()` from `lib/healthie/client.ts`.** Don't `fetch()` Healthie's GraphQL endpoint directly — you'll forget the AuthorizationSource header.
   2. **Define a Zod schema for the response shape.** Healthie returns nested objects; the schema gates Zod-validates each shape and blocks shape drift. Match the GraphQL mutation/query nesting (e.g. `{ user: {...} }` for createClient — see Gotcha #4).
   3. **Pass the `resultKey` parameter.** It must match the top-level GraphQL field name (e.g. `'createClient'` for `mutation { createClient(...) { user { ... } } }`).
   4. **Handle `HealthieApiError` explicitly.** Three failure modes: HTTP error (statusCode populated), GraphQL errors[] (graphqlErrors populated), validation failure (just message). Surface to admin UI; never re-throw raw.
   5. **Log without PHI.** Names, DOB, email, address, biomarker values, form answers are PHI. Healthie internal IDs (user_id, appointment_id) are OK. See gotcha #6.
   6. **Wrap idempotent reads with `lib/resilience.ts` retry.** Mutations like `createClient` MUST NOT auto-retry — use the idempotent `ensureHealthiePatient` pattern (lookup-by-email-then-create) instead.
   7. **NUMERIC coercion.** `@vercel/postgres` returns NUMERIC columns as strings; coerce with `Number()` before forwarding to Healthie or doing arithmetic.
   8. **`skipped_email: true` on every client create.** CULTR owns the welcome-email funnel.
